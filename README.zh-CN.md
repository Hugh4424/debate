## 一句话讲清痛点

你的 AI 审查代码，发现问题，然后*同一个* AI 决定哪些发现该采纳。

> **AI 审查员：** 发现 #3 —— 这个方案有风险。
> **AI 作者：** ✅ 没事，我的设计没问题。

它当然驳回了。它在给自己的作业打分——写的时候有啥盲区，裁的时候还是那些盲区。那个绿色勾是**演戏。**

`debate` 用一条规则解决它：

> **做决策的，永远不许给自己放行。**

每一条有争议的发现都送上五方法庭。独立子代理分饰控辩各方，互相质疑。法官（Claude）被**全程禁言**——不下场辩护，不偏袒任何一方，只综合幸存论点给出裁决。

## 看个真实案例

> *"我们在做一个公开 API。Claude 提议用无过期时间的长效密钥。审查员在安全立场上提出了异议。谁是对的？"*

**输入** → [`examples/sample-input.md`](examples/sample-input.md) — 三条 Claude 决策、三条审查发现，含 MR-2 分类。

**输出** → [`examples/sample-verdict.md`](examples/sample-verdict.md) — 一轮完整辩论后的裁决表：

| 提案 | 裁决 | 决定性论点 |
|---|---|---|
| 长效密钥，无过期时间 | **部分采纳** | 丁的第三条路（90天自动续期）幸存；"Stripe 从不轮换"论点被丙击败 |
| 信任内部 header，不做二次校验 | **有条件采纳** | 合并前须补 ADR——乙的顾虑在于未文档化的假设，而非机制本身 |
| 暂不加 scope 字段 | **采纳** | 丙丁均确认符合 YAGNI；乙未提异议 |
| 新增 `last_used_at` 字段 | **待定——用户决定** | 双方论点均幸存；丁标记了潜在的策略影响 |

完整辩论记录落盘至 `debate/round-1/` 供审计。[查看完整示例 →](examples/sample-verdict.md)

## 你能得到什么

| 你的情况 | 平时会发生啥 | 用了 debate |
|---|---|---|
| AI 自审自判 | 它替自己辩护，盲区照旧 | **五个独立代理**对两方同时开炮 |
| 对自己设计的偏袒式「通过」 | 没人发现 | Claude **被禁止下场辩护** |
| 发现被轻描淡写驳回 | 审查流于形式 | 每条发现必须**在质疑中幸存** |
| 没有 Agent Teams | 技能无法运行 | **自动降级**单人三档，同格式输出 |
| 小分歧，不值得开庭 | 无谓的对抗开销 | 只有**方向级分歧**才触发 |

## 快速上手

`debate` 独立可用——给它两方提案，直接跑。不需要任何流水线前置。

**1. 安装技能：**

```bash
git clone https://github.com/Hugh4424/debate ~/.claude/skills/debate
```

**2. 在 Claude Code 中调用：**

```
/debate
```

提供你的提案或审查发现，技能接管后续流程。

**输出是一张裁决表**——每条有争议的提案一行，附裁决结果（采纳/部分采纳/驳回）和决定裁决的幸存论点。完整辩论记录落盘供审计，你不需要翻它。

**可选——开启完整五方模式：**

```bash
export CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS=1
```

没有这个环境变量，技能自动降级为单人三档裁决——输出格式相同，没有多代理开销。独立模式是推荐的起点。

## 凭什么相信这个裁决

裁决太好造假了。三道约束让这个真的算数：

- **Claude 的决策是被告，不是法官。** Claude 的原始设计决策和审查员的发现对等上庭受审，谁都不是默认基准。

- **法官全程禁言。** Claude 只在 Phase 3 作为综合裁决的法官——在辩论阶段不得为任何一方出声辩护。它一旦开始替自己的方案辩护，整件事的意义就垮了。

- **每条裁决必须引用幸存论点。** 「采纳」或「驳回」必须说明哪些论点经过交叉质疑后幸存、哪些被驳倒。无幸存论点 → 无效裁决。

## 五方结构

五个角色，四个由独立子代理扮演：

| 角色 | 立场 | 由谁扮演 |
|---|---|---|
| **甲队** — A 方代言人 | 论证 A 方每条决策成立 | 独立子代理 |
| **乙队** — B 方代言人 | 论证 B 方每条提案该被采纳 | 独立子代理 |
| **丙队** — 杠精 | 对甲乙**所有**提案逐条否定 | 独立子代理 |
| **丁队** — 影响评估+搅局者 | 评每条代价/收益 + 揪隐藏前提和虚假共识 | 独立子代理 |
| **Lead 法官** | 辩论期间禁言；综合幸存论点给出最终裁决 | Claude 本人 |

**核心铁律**：甲乙对等——两方都是**被告**，谁都不是默认基准。丙丁对两方同等猛攻。

## 什么时候触发

不是每个分歧都需要开庭。`debate` 只对**方向级分歧**触发，按 MR-2 分类：

| 类型 | 示例 | 触发 debate？ |
|---|---|:---:|
| 问题定义 | "我们解决的是对的问题吗？" | ✅ 触发 |
| 范围/优先级 | "这件事该现在做吗？" | ✅ 触发 |
| 需求理解 | "规格文档说的到底是什么？" | ✅ 触发 |
| 实现细节 | "用哪个库、变量叫什么名字？" | ❌ 不触发 |

纯实现层的分歧由主代理直接处理。只有方向级分歧上法庭。

## 四阶段流程

```
Phase 0  立案分派    法官整理案卷，把提案分配到甲/乙两方
Phase 1  独立陈述    四队并行独立写立场书，禁止互通
Phase 2  法庭辩论    丙丁质疑甲乙，甲乙逐条回应（每条≤200字），最多2轮
Phase 3  综合裁决    法官综合幸存论点，逐条裁决
```

## 踩过的坑都焊进去了

- **法官禁言规则。** 实测：不加这条，Claude 在 Phase 3 持续轻描淡写地驳回挑战自己设计的发现（理由是"这个方案本来就是我这样设计的"）。加了禁止 Phase 2 辩护的限制后，这个模式几乎消失。

- **双方对等被告。** 早期版本把一方放在主位——法庭论的是"对方发现成不成立"，而不是"两方决策谁更对"。结果只是把橡皮章从采纳盖到了驳回。让两方对等上庭才是真正的解法。

- **两轮上限。** 三轮辩论增加的新幸存论点几乎为零，但多花了约 40% 的 token。上限定在两轮。

## 什么时候不用它

**以下情况不用调用：**
- 所有分歧都是实现层的（命名、库选型、代码风格）
- 还没有提案或审查发现——`debate` 需要已有立场作为输入，不负责生成
- 一方明显正确、没有合理反驳空间

它在存在**方向级分歧**、你需要一个不受"谁上下文写得更多"影响的公正裁决时，才真正值回票价。
